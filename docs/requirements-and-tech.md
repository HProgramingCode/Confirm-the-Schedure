# 工場シフト確認アプリ — 要件・技術まとめ

会話および [.cursor/skills/requirements-tech-spec/SKILL.md](../.cursor/skills/requirements-tech-spec/SKILL.md) の章立てに沿って整理した。**MVP のみ**を実装スコープとし、班別などの絞り込みは追加開発とする。

---

## 1. 概要（背景・目的・成功指標）

### 1.1 背景

- 工場ではシフトが**紙（年1回配布）**で配られており、従業員が掲示を確認している。
- **毎年同じ休み・並びとは限らない**。新しい紙が配られるタイミングで、参照すべき表が入れ替わる。
- 紙の内容を **Web で見える化**し、**カレンダー**で確認しやすくしたい。
- **プロジェクトの位置づけ（確定）**: **個人開発**であり、**会社の公式システム・人事・勤怠等とは連携しない**。会社から配布された紙の情報を、**本人・職場の仲間が任意に**本アプリへ登録し参照する用途に限定する（公式性・監査は求めない）。

### 1.2 紙フォーマットの前提（確定）

- **レイアウトは毎年同じ形式**で配布される（行・列の構成、記号の置き方が年度をまたいでもほぼ固定）。取り込みでは**テンプレート前提**の補助（固定グリッド UI、座標・切り出しの再利用、OCR の前提画像）がしやすい。
- **1・2・3・休みなどの中身は年度ごとに変わり、月ごとにも異なる**（同じ記号でも並び・パターンはデータ）。アプリは**ルールで休みを再計算しない**。**版（年度）× 月**ごとに紙の内容をそのまま保持する。

### 1.3 目的（MVP）

1. **対象年度・月**の **A / B / C / D 各行**について、各日の **1・2・3・休** をシステムに反映する（**アプリは割り振りを決めない**。紙の内容の再現）。**初期リリースは手動入力**を優先し、**ある程度完成後**に画像からの自動取り込みを追加する（§6.3）。
2. **ユーザー登録**後、**本人のみ**が自分と **A〜D のいずれか1つ** を対応付ける（会社の割当と揃えるのは**本人の自己申告**。誤選択リスクは §7.2）。
3. **カレンダー**で、**自分の列**を初期表示し、**同じアルファベットの他ユーザー**を**本名のフルネーム**で分かるようにし、**他の A〜D を選択**してその列も閲覧できる。
4. **UI/UX**: **紙のレイアウトを再現する必要はない**。**業務利用・全年齢**を想定し、**オシャレな装飾やトレンド表現より**、**どの年代でも見やすい文字サイズ・コントラスト**、**迷わない導線**、**誤タップしにくい操作**を優先する（地味で構わない）。利用者は**スマートフォン利用を主**とし、**狭い画面で崩れない**こと。PC は**スマホ画面を広げた形**でもよい（レスポンシブ1系統でよい）。コンポーネントは **shadcn/ui 利用可**（見た目は業務向けにトーンを抑えて調整 §6.3・[stack.md](./stack.md)）。

### 1.4 成功指標（案・未確定）

- **MVP**: 反映した月について、紙と**セル単位で矛盾がない**（運用でダブルチェックできる状態）。
- 数値目標（利用率など）は**未定義**。必要になったら別途定義する。

### 1.5 用語

| 用語 | 意味 |
|------|------|
| **列（アルファベット）** | A / B / C / D のいずれかの行。会社が各人をどれかに割り当てる。 |
| **セル値** | 各日に対する **1 / 2 / 3 / 休**（紙に印刷されている記号の意味は現場定義）。 |
| **版（年度）** | 年1回の配布に対応するデータのまとまり。MVP でも「どの配布分か」を識別できるようにする。 |

## 2. スコープ（含む／含まない）

### 2.1 MVP に含む

- **シフトマスタ登録**: **指定した版・年月**の **A〜D × 各日** のセル値を登録・更新する手段。**フェーズ1（MVP 初期）**は**手動入力**（表形式でも日別リストでもよいが、**スマホで操作しやすい**こと）。**フェーズ2**で画像からの取り込みを追加（§6.3）。
- **版（年度）**: 少なくとも**現在有効な版**と、必要に応じ**過去版の参照**を想定した識別子（年1回更新に対応）。
- **ユーザー**: 登録・ログイン（方式は技術選定）。
- **紐付け**: ユーザーと **assignedLetter ∈ {A,B,C,D}**（**1ユーザー1文字**）。会社の割当と揃える運用。
  - **年度で所属文字が変わり得る**場合に備え、データ上は **版（edition）ごとに紐付け**を持てる形を推奨（MVP で複数版を扱う場合に整合しやすい）。
- **閲覧**: 月次カレンダー（**その月の実日数**に合わせて表示。31日・2月などは**紙に載っている通り**を保存する）。
- **UI/UX**: **モバイルファースト**。**自分の列**・**同じ文字のユーザー（フルネーム）**・**A/B/C/D の切替**で他列表示。レイアウトは紙に似せる**必須はない**（§1.3）。**装飾より可読性・操作性**（§1.3）。

### 2.2 MVP に含めない（追加開発へ）

- **班別・グループ別の絞り込み**（検索・フィルタ）。A〜D のみで完結させたのち拡張する。
- **1・2・3・休の自動再計算・最適化**（ルールエンジン）。
- **代行・振替申請**などのワークフロー。
- **プッシュ通知・メール通知**（新年度反映の告知など）は将来検討。
- **前年との差分表示**、**PDF/印刷**は将来検討。

### 2.3 利用者

- **主利用者**: 工場の従業員（自分のシフト・同列メンバー・他列の確認）。**A〜D の紐付けは本人のみ**が設定する。
- **データ投入**: **個人開発**のため、版・月次グリッドを編集できる**管理者ロール**（開発者または信頼できる少数）を想定。会社からのデータ連携は**ない**（§1.1）。

## 3. 機能要件

### 3.1 管理者（データ投入）

| ID | 要件 |
|----|------|
| A-1 | **版**を登録できる（例: 年度ラベル、備考）。 |
| A-2 | **年月**を指定し、その月の **A〜D 各行**について、**日1〜月末**までのセル値（1/2/3/休）を登録・更新できる。 |
| A-3 | **フェーズ1**: 手動でグリッド入力。**フェーズ2**: **写真（画像）**から取り込み（**レイアウト固定を活かした座標ベースのセル切り出し**＋ OCR 等の組み合わせを想定。詳細は実装時に `docs/stack.md` 等で確定）。取り込み後も**手動修正**可能であること。 |
| A-4 | 登録内容が、紙の当該月と**照合可能**であること（画面上で一覧確認できる）。 |

### 3.2 一般利用者

| ID | 要件 |
|----|------|
| U-1 | **ユーザー登録**および**ログイン**ができる。 |
| U-2 | ログイン後、**本人のみ**が**自分の assignedLetter** を設定・変更できる（管理者による代設定は**しない**）。 |
| U-3 | **月**を選び**自分の列**のカレンダーを見られる。**版が複数ある場合のデフォルト表示は最新版**（過去版は明示的に選択）。 |
| U-4 | **同じ assignedLetter** の他ユーザーを**本名のフルネーム**で一覧等に表示する（当面の方針。**未確定**: 後から表示名のみへの変更要否）。 |
| U-5 | **A / B / C / D** を切り替え、自分以外の列の当該月カレンダーを閲覧できる。 |

### 3.3 データ要件（論理）

- **ScheduleEdition**: 版ID、表示名（例: 2025年度）、作成日時など。
- **MonthlyShiftGrid**: editionId、year、month、行 A〜D、各 **day（1..その月の日数）** に cellValue ∈ {1,2,3,休}（紙と同じ日付範囲）。
- **User**: 認証に必要な識別子、**本名フルネーム**（当面は同列表示に使用 §3.2 U-4）など。
- **UserLetterBinding**: userId、editionId、assignedLetter（MVP から複数版を扱うなら必須に近い）。

### 3.4 画面・操作（MVP）

- **管理者**: 版一覧／作成、月次データの編集（**スマホでも破綻しない** UI。表・分割フォーム等は実装で選択）。フェーズ2 で画像取り込み導線を追加。
- **利用者**: ログイン・**本人による**紐付け設定、カレンダー／シフト閲覧（**デフォルト最新版**・月・列切替、同列メンバー＝**フルネーム**）。**タップしやすいコントロール**・**横スクロール時も意味が分かる**表示を意識する。

## 4. 非機能要件

| 項目 | MVP の方針 |
|------|------------|
| セキュリティ | ログイン必須。HTTPS。シフト・**フルネーム**は**任意利用の個人開発アプリ**内で扱う（会社システムと非連携 §1.1）。**管理者／一般**の分離は**簡易**でよい（厳密な多段ロールは未確定）。 |
| 可用性 | **未定義**（個人開発・小規模想定なら単一環境でも可と記載）。 |
| 性能 | 同時アクセス規模は**未定義**。月次1画面のデータ量は小さい。 |
| バックアップ | **未定義**（本番運用時に方針決定）。 |
| 監査 | MVP では**必須としない**。「誰がいつ版を更新したか」は将来拡張で可。 |
| UI/UX・アクセシビリティ | **スマホファースト**（§1.3）。**業務系・全年齢向け**: 文字はやや大きめ、余白とコントラストを確保、**不要なアニメーション・遊び心の UI は避ける**。PC は同一レイアウトの拡大表示で可。**タップ領域・フォーカス**など、可能な範囲で [WCAG](https://www.w3.org/WAI/standards-guidelines/wcag/) を意識（厳格な準拠までは MVP で強制しない）。実装時は [.cursor/skills/design-ui/SKILL.md](../.cursor/skills/design-ui/SKILL.md) を参照。 |

## 5. 制約・前提

- **紙が正**: 1/2/3/休 の意味・勤務時間の解釈は**現場の説明に依存**。アプリは記号を**保存・表示**する（意味付けロジックは MVP では持たない）。**パターンは月ごとに異なる**ため、**計算で休みを導出しない**（§1.2）。
- **レイアウト固定**（§1.2）により、画像取り込み時は**テンプレに沿った OCR 補助**が現実的。手動入力時は**データを正しく入れられれば**表示形式は自由（§1.3）。セルと画像の**自動対応のみ**に依存しない（手動修正 UI は必須に近い）。
- 現場では**会社が各人を A〜D のいずれかに割り当てる**が、本アプリでは**会社と連携しない**（§1.1）。Web 上の紐付けは**本人が自己申告**するのみ。誤りは**本人・利用コミュニティ側の運用**で防ぐ（台帳との自動突合は**ない**）。
- プロジェクト直下の [.cursor/rules/nextjs-stack.mdc](../.cursor/rules/nextjs-stack.mdc) に、**Next.js / DB / 認証 / クライアント状態**のメモがある。実装の入口は **[README.md](../README.md)**、技術の詳細は **[stack.md](./stack.md)** を正とする。

## 6. 技術選定

### 6.1 スタック（論点・前提）

| 論点 | 候補・検討 |
|------|------------|
| フロント／BFF | Next.js（App Router）— ルール上の前提。 |
| DB | Turso（本番）／SQLite（ローカル）— ルール上の前提。 |
| 認証 | NextAuth 等 — ルール上の前提。 |

### 6.2 画像から表への取り込み（Next.js での実現可否と候補）

**結論**: Next.js 単体で「アップロード → 処理 → DB」まで**実現可能**。難所は「画像上のどのセルが A 行の何日目か」の自動対応で、**レイアウト固定**（§1.2）でも撮影ゆらぎは残るため、**手動確認・修正 UI** を MVP に含めるのが安全。

**処理の流れ（例）**

1. ブラウザで画像選択 → **Route Handler または Server Action** でサーバへ送信（またはクライアントで OCR し結果のみ送信）。
2. OCR／Vision で文字・レイアウト候補を取得。
3. **A〜D × 各日**のグリッドにマッピング（ルール・テンプレ・人の修正）。
4. DB に保存（`MonthlyShiftGrid`）。

**候補の比較（コスト感の目安）**

| 方式 | 概要 | 無料寄り・備考 |
|------|------|----------------|
| **Tesseract.js** | ブラウザまたは Node で WASM OCR。API キー不要。 | **追加従量課金なし**。サーバレス上の Node で重い画像を回すとタイムアウトしやすいので、**クライアントで OCR → テキストだけサーバ送信**も選択肢。表構造の自動推定は**弱め**。 |
| **Google Cloud Vision API** 等 | サーバから画像を送りテキスト検出（ドキュメント向け機能でレイアウト情報が取れる場合あり）。 | 公式料金に**月あたりの無料ユニット**あり。**年に数回〜十数回**程度なら無料枠内に収まりやすい規模（機能・単位は都度 pricing で確認）。GCP 登録・**画像の外部送信**・規程確認が必要。 |
| **手入力メイン** | 画像を横に表示し、管理者がグリッドに直接入力。 | **コストゼロ・最も確実**。年1回×複数月でも運用で許容されやすい。 |
| **ハイブリッド（推奨検討）** | 上記いずれかで下書き → **必ず画面上で修正して確定**。 | 誤認識リスクを抑えつつ、無料〜低コストで回しやすい。 |

**取り込み回数の前提**: 画像解析の実行回数は**年に多くても十回未満**想定。無料枠の小さい手段でも足りることが多く、精度優先でクラウド Vision を少量使っても**課金は極小**になりやすい（正確な見積は各社料金表で確認）。

**フェーズ2 での実装イメージ（確定方針・詳細は未実装）**

- 紙は**レイアウト固定**（§1.2）のため、**画像上の座標（または正規化した座標）でセル領域を定義**し、切り出して **OCR** する方式が有力（いわゆる座標ベース／テンプレマッチに近いアプローチ）。必要に応じ §6.2 の Vision・Tesseract と組み合わせる。
- 取り込み後は**手動修正 UI** を維持する。

### 6.3 決定と理由（取り込み・表示）

| 項目 | 決定 |
|------|------|
| MVP 初期のデータ入力 | **手動**（**スマホ向けに使いやすい** UI。紙と同じ見た目は不要）。 |
| 画像の自動取り込み | **後追い実装**（アプリがある程度完成してから）。**座標ベースのセル切り出し＋ OCR 等**を第一候補とする。 |
| 紐付け | **本人のみ**。 |
| 同列メンバー表示 | **本名フルネーム**（当面）。 |
| 版のデフォルト | **最新版**。 |
| UI/UX | **モバイルファースト**・**業務向け（装飾より可読性・全年齢）**・**shadcn/ui 可**（トーンは抑える）。紙レイアウトの再現は**不要**（§1.3・§4・[stack.md](./stack.md)）。 |
| 会社連携 | **なし**（個人開発・配布物の見える化のみ §1.1）。 |

- スタックの詳細は **[stack.md](./stack.md)** に記載した。画面・DB のたたき台は **[mvp-design.md](./mvp-design.md)**。

### 6.4 見送り・保留

- 班別マスタ、通知、差分表示、監査ログは **MVP 外**。

## 7. 未決・リスク・次のアクション

### 7.1 未決事項（要決定）

- **成功指標の数値**（利用率など）は引き続き未定義。
- **通知**（新年度反映のお知らせ等）の要否。
- **管理者ロール**を複数段に分けるか（現状は「簡易でよい」とするが、厳密な権限モデルは未確定）。
- **同列表示をフルネーム以外**（表示名のみ等）に変えるかは、後から要検討（当面はフルネーム §6.3）。

### 7.2 リスク

- **OCR 誤認識**（フェーズ2）→ **手動修正 UI** を必ず残す（§6.2・§6.3）。
- **本人が誤った A〜D を選択** → **会社システムと非連携**のため**自動突合はない**。職場内の口頭・紙との照合に依存（§1.1・§5）。
- **月ごとにパターンが変わる**（§1.2）ため、前年データのコピーミスや、別月の取り違えに注意（版・年月の明示 UI）。
