## レイヤー構成

### 1. プレゼンテーション層（`src/app/`）

- **Server Component** を基本とし、ブラウザ API・`useState`・イベントハンドラが必要な場合のみ `"use client"` を付与
- ルーティングは App Router のファイルシステムベース
- UI は Tailwind CSS + shadcn/ui（業務向けにトーンを抑えた設定）

### 2. ビジネスロジック層（`src/app/actions/`）

- **Server Actions** でフォーム送信・データ変更を処理
- Zod によるバリデーション
- 認可チェック（`auth()` でセッション取得後に `isAdmin` 等を検証）

### 3. データアクセス層（`src/lib/db/`）

- **Drizzle ORM** でスキーマ定義とクエリを TypeScript で記述
- DB 接続は `@libsql/client` を使用
- マイグレーションは `drizzle-kit` で管理

### 4. 認証層（`src/auth.ts`）

- **Auth.js v5**（next-auth）を使用
- Credentials プロバイダ（メール + パスワード）
- JWT セッション（30日間有効）

## 主要な設計判断

| 判断項目 | 採用方針 | 理由 |
|---------|---------|------|
| レンダリング | Server Component 優先 | パフォーマンス・SEO・バンドルサイズ削減 |
| データ取得 | Server Component 内で直接 DB アクセス | API 層を減らしシンプルに |
| データ変更 | Server Actions | フォーム処理の型安全性・リダイレクト連携 |
| 認証 | JWT セッション | 小規模・DB セッション管理の省略 |
| 状態管理 | Zustand（必要時のみ） | URL・サーバーが正とし、クライアント状態は最小限 |
| スタイリング | Tailwind CSS | 既存方針・ユーティリティファースト |

## 改修時の注意点

1. **Server Component と Client Component の境界を意識する**
   - `"use client"` は必要最小限のコンポーネントに限定
   - 詳細は [frontend.md](./frontend.md) を参照

2. **Server Actions の認可チェックを必ず行う**
   - `auth()` でセッション取得後、`isAdmin` 等のフラグで権限を検証
   - 詳細は [backend.md](./backend.md) を参照

3. **DB スキーマ変更時はマイグレーションを作成する**
   - `npm run db:generate` → `npm run db:migrate`
   - スキーマは `src/lib/db/schema.ts` で一元管理

4. **既存のコーディング規約・ファイル構成に従う**
   - 詳細は [folder-structure.md](./folder-structure.md) を参照
