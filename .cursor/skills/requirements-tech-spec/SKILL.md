# 要件定義〜技術選定（洗練・再編成）

## 目的

- 断片的な要件・技術メモを **構造化**し、**重複と矛盾**を減らす。
- **足りない情報は質問**で埋め、埋まったあと **読みやすい形に再編集**する。
- 成果は **`docs/` 配下の1ファイル**（例: `docs/requirements-and-tech.md`）にまとめる。既存の命名があればそれに合わせる。

## 進め方（フェーズ）

### 1. 入力の把握

- ユーザーが書いた箇条書き・会話・メモをそのまま受け取り、**確定事実 / 推測 / 未確定**にラベル付けする（推測は明示）。

### 2. ギャップを質問で埋める

次のブロックごとに、**未記載・曖昧なら質問を列挙**（はい・いいえ／選択／自由記述で答えやすい形）。

| ブロック | 含めること |
|----------|------------|
| 背景・目的 | なぜ今やるか、成功の定義 |
| スコープ | やること／やらないこと、対象ユーザー |
| 機能要件 | 必須ストーリー、優先度 |
| 非機能 | 性能、可用性、セキュリティ、運用、コスト感 |
| 制約 | 期限、人員、既存システム、法規 |
| 技術前提 | 既に決まっているスタック、変えられないもの |
| 技術選定の論点 | 候補、比較軸、決め手にしたいこと |

**質問は一度に多すぎない**（目安5〜10個）。回答を受けてから次の質問へ。

### 3. ドラフト統合

回答と既存メモを統合し、次の **章立てドラフト**を Markdown で出す。

```markdown
# （プロジェクト名）要件・技術まとめ

## 1. 概要（背景・目的・成功指標）
## 2. スコープ（含む／含まない）
## 3. 機能要件
## 4. 非機能要件
## 5. 制約・前提
## 6. 技術選定
### 6.1 候補と比較
### 6.2 決定と理由
### 6.3 見送り・保留
## 7. 未決・リスク・次のアクション
```

### 4. 洗練パス（完了後の再編集）

ユーザーが「ここまでで要件・技術は固めた」と言ったら、**編集モード**に切り替える。

- **用語を統一**（同義の言い換えを1つに）。
- **重複段落をマージ**、**見出し階層**を整理。
- **判断と事実を分ける**（「〜とする」は決定、「〜である」は事実）。
- **非機能を測定可能なら数値化**（できなければ「未定義」と明記）。
- **技術選定**は「選んだもの・理由・捨てたもの・トレードオフ」を1箇所に集約。
- 最後に **目次互換の見出し**と、**未決事項だけの短いリスト**を付ける。

### 5. ルール・スキルとの整合

- プロジェクトに **`docs/stack.md`** や README があれば、技術選定と**矛盾しない**よう揃える。
- 確定した運用（タスク・日記・ジュニア向けメモ）は既存の `workflow-progress` 等に任せ、**本スキルは「要件・技術の本文」に専念**する。

## 注意

- **推測で要件を補完しない**。不明は質問か「未確定」として残す。
- **トークン・秘密・個人名**は文書に書かない。
